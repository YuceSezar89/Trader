### 1. Redis MTF Cache Entegrasyonu

#### Dosya: `utils/redis_client.py`
```python
# Yeni fonksiyonlar eklenecek:
- set_mtf_klines(symbol, timeframe, df)
- get_mtf_klines(symbol, timeframe)
- set_mtf_indicators(symbol, timeframe, indicators)
- get_mtf_indicators(symbol, timeframe)
- flush_mtf_cache(symbol, timeframes)
```

#### Cache Key Yapısı:
```
Mevcut: live_kline_data:BTCUSDT
Yeni:   live_kline_data:BTCUSDT:1m
        live_kline_data:BTCUSDT:5m
        live_kline_data:BTCUSDT:15m
        hot_indicators:BTCUSDT:5m
        hot_indicators:BTCUSDT:15m
```

#### TTL Stratejisi:
- 1m data: 1 saat TTL
- 5m data: 4 saat TTL  
- 15m data: 24 saat TTL
- Indicators: 30 dakika TTL

### 2. LiveDataManager MTF Desteği

#### Dosya: `live_data_manager.py`
```python
# Yeni özellikler:
- self.mtf_buffers: Dict[str, Dict[str, pd.DataFrame]]
- _update_mtf_data(symbol, kline_data)
- _aggregate_and_cache_mtf(symbol)
- _process_mtf_signals(symbol)
```

#### İş Akışı:
1. 1m bar kapanışında tetiklenir
2. 1m verisi buffer'a eklenir
3. 5m ve 15m aggregation hesaplanır
4. Tüm TF'ler Redis'e cache'lenir
5. Her TF için sinyal işleme başlatılır

## ⚠️ RİSKLER VE ZORLUKLAR

### Teknik Riskler
- **Memory Kullanımı**: 5x artacak (100 symbol × 3 TF = 300 buffer)
- **CPU Overhead**: Aggregation hesaplamaları
- **Redis Memory**: Cache boyutu artacak
- **Database Load**: Daha fazla yazma işlemi

### Çözüm Stratejileri
- Circular buffer kullanımı (son 1000 bar)
- Intelligent caching (hot/cold data)
- Batch processing optimization
- Memory monitoring ve alerting

### Performance Hedefleri
- Memory kullanımı: <2GB Redis
- Aggregation süresi: <100ms per symbol
- Cache hit rate: >90%
- Database yazma: <1000 insert/dakika
